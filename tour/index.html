Station 03 · Plan

/we:refine — turn an idea into a story.

You bring a sentence, a ticket key, a vague intent. /we:refine asks the questions that turn it into something you can actually build — and writes the plan that the next phase will read.

claude code · /we:refine
/we:refine "add Stripe checkout to settings"
Who pays — only the workspace owner, or any member?
workspace owner only
Which plans should they be able to pick?
Bond / Companion / Soulmate (existing 3 tiers)
What happens to the existing free-trial users on upgrade?
prorate the remaining trial days against the first invoice
Ticket WA-1234 created · plan written to docs/plans/WA-1234-plan.md
plan now contains: context, 4 acceptance criteria, 3 phases, test approach, security review, design decisions (incl. rejected alternative: full-stripe-checkout-redirect)

It asks, you decide.

The shape of the questions is determined by what's *missing*, not a fixed template. Quiet stories pass quickly; contentious ones get more questions.

Two outputs.

A minimal ticket („As X I want Y so that Z") plus a detailed plan file with context, AC, phases, tests, security review, and *rejected alternatives*. The next skill reads the plan and understands intent — not just spec.

When you want two voices on it.

For contentious stories where one perspective isn't enough, use /we:meet refinement instead. Convenes a small council (PO + Architect) for tension before the plan crystallizes. Then hands off to /we:refine once aligned. Next station →

Station 05 · Build · the autonomous pipeline

/we:story — hand it a ticket, watch it ship.

Nine steps from worktree to PR. Sequential where order matters, parallel where it doesn't. Checkpoints between every step — interruption-safe. By the time you handed it a ticket, you already decided to build it; /we:story doesn't ask permission for each phase.

01

Git prep

Worktree, branch, ticket → In Progress.

02

Develop

Implement plan phase by phase. TDD where it fits. Auto-fix after each phase.

03
Blocking

AC verify

Every acceptance criterion checked with concrete evidence. No item passes without a file:line citation.

04

Simplify

Remove dead code, simplify expressions, reuse existing helpers. (via code-simplifier plugin)

05
Parallel

Quality gates

Code review + static analysis + tests. All three subagents fan out in a single message — concurrent.

06

Docs

doc-architect agent proposes diffs. Never writes autonomously — every change is reviewable.

07

PR

/we:pr verifies all three quality-gate checkpoints first. Will not create a PR with failing gates.

08
Loop

CI fix

Collect ALL findings, fix in ONE commit, push ONCE. Max 3 cycles. CodeRabbit threads must be resolved.

09

Hand off

Ticket → In Review. Never „Done" — that move stays with you.

Station 06 · How it stays correct

Discipline you can feel — not theatre.

An autonomous pipeline is only as trustworthy as its boundaries. These four mechanisms are why a green /we:story doesn't surprise you on review.

AC verification Blocking

Step 3 will not pass an acceptance criterion without concrete evidence — file:line, test name, commit hash. A handwave doesn't qualify. If evidence can't be produced, the pipeline halts and asks.

Parallel quality gates Concurrent

Step 5 dispatches code-review, static analysis, and tests as three subagents in a single message. Concurrent fan-out, single join. Three findings come back together — not three serial waits.

CI batch-fix Max 3 cycles

When GitHub CI surfaces findings, the loop collects all of them, fixes all of them in ONE commit, pushes ONCE. Three cycles is the cap before /we:story stops and asks. Anti-thrash by design.

Checkpoint + circuit breaker Resumable

SQLite checkpoints at every phase — interrupt the session and resume with the same command, no lost work. Plus: three failures in the same phase trip the circuit breaker. Stop and ask, don't thrash. ~/.claude/weside/orchestration.db

Station 07 · Improve how we work

/we:sm — the Scrum Master lens.

The pipeline ships stories. Once in a while something flatlines — a CI cycle that won't converge, a refinement that took five rounds, an AC that should have been clearer. /we:sm reflects with the Scrum Master lens to ask: what did we learn? what changes in how we work? The plugin improves itself, through its own meta-skill.

01 · TRIGGER

Friction surfaces

Pipeline thrashes on the same gate, a refinement keeps cycling, a pattern shows up across stories. You feel it, or the circuit breaker tells you.

→
02 · REFLECT

/we:sm convenes

Scrum Master reads the situation — recent commits, plan files, ticket history, what the council said and what the gates flagged. Proposes one focused change, not a list.

→
03 · LAND

Plugin improves itself

The change lands in .claude/rules/, in a plan template, or in a skill's behaviour. Next story benefits — the workflow you ship with tomorrow isn't the one you shipped with today.

Station 08 · Deliver — you

What you get. What stays yours.

By the time /we:story hands off, the PR is in a known-good state. The remaining work — judging it, merging it, closing the ticket — is human work. By design.

What /we:story produces

  • All acceptance criteria implemented and evidenced
  • Tests passing (unit + relevant integration)
  • Code reviewed (code-reviewer agent + CodeRabbit on GitHub)
  • Doc diffs proposed and applied
  • CI green (with CodeRabbit threads resolved)
  • Ticket sitting in „In Review"

What stays yours

  • Reading the PR with your eyes
  • Judging the change — is this still what we wanted?
  • Merging the PR
  • Closing the ticket
  • Communicating the change to stakeholders
  • The decision to ship
Claude never merges PRs or closes tickets. Not a feature flag, not a configuration — a property of the design. Autonomy ends at the gate where consequence begins.
