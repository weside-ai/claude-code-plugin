Station 02 · The big picture

Station 03 · Vision · Saga-level

/we:meet vision — where are we going?

A vision meeting convenes when direction needs alignment — a new market, a strategic pivot, a brand bet, a year-shaping decision. The horizon is weeks to months. The output isn't code — it's clarity, captured as a direction document.

Station 05 · Refinement · Story-level

Story-level — two paths to the same plan.

When a story needs scoping before development, you have two paths. The solo path is fast and works for routine refinements. The council path adds tension before the plan crystallizes — for the harder calls.

/we:refine "<idea>"
Solo · interactive Q&A
Claude asks the missing questions; you answer. Quiet stories pass quickly; contentious ones get more questions. Best for routine refinements.
idea → Q&A → ticket + plan
/we:meet refinement
Council · two perspectives
PO and Architect deliberate on scope and acceptance before the plan crystallizes. After deliberation, the meeting hands off to /we:refine.
topic → council → /we:refine → ticket + plan

Station 06 · Build · the autonomous pipeline

/we:story — hand it a ticket, watch it ship.

Nine steps from worktree to PR. Sequential where order matters, parallel where it doesn't. Checkpoints between every step — interruption-safe. By the time you handed it a ticket, you already decided to build it; /we:story doesn't ask permission for each phase.

01

Git prep

Worktree, branch, ticket → In Progress.

02

Develop

Implement plan phase by phase. TDD where it fits. Auto-fix after each phase.

03
Blocking

AC verify

Every AC checked with concrete evidence. No item passes without a file:line citation.

04

Simplify

Remove dead code, simplify expressions, reuse helpers.

05
Parallel

Quality gates

Code review + static + tests. Three subagents in a single message — concurrent.

06

Docs

doc-architect proposes diffs. Never writes autonomously.

07

PR

Verifies all three quality gates first. Won't create a PR with failing gates.

08
Loop

CI fix

Collect ALL findings, fix in ONE commit, push ONCE. Max 3 cycles.

09

Hand off

Ticket → In Review. Never „Done" — that move stays with you.

AC Verify · Blocking
No claim without citation
Step 3 will not pass an AC without file:line evidence. Halts and asks if it can't.
Gates · Parallel
Three reviewers, one wait
Step 5 dispatches review + static + tests in a single message. Concurrent fan-out.
CI Loop · Bounded
Batch-fix, max 3 cycles
Findings collected, fixed in one commit, pushed once. After 3 cycles: stop & ask.
Resume · Safe
Checkpoint + circuit breaker
SQLite checkpoints survive interruption. 3 same-phase failures trip the breaker.

Station 07 · Improve how we work

/we:sm — the Scrum Master lens.

The pipeline ships stories. Once in a while something flatlines — a CI cycle that won't converge, a refinement that took five rounds, an AC that should have been clearer. /we:sm reflects with the Scrum Master lens to ask: what did we learn? what changes in how we work? The plugin improves itself, through its own meta-skill.

01 · TRIGGER

Friction surfaces

Pipeline thrashes on the same gate, a refinement keeps cycling, a pattern shows up across stories. You feel it, or the circuit breaker tells you.

→
02 · REFLECT

/we:sm convenes

Scrum Master reads the situation — recent commits, plan files, ticket history, what the council said and what the gates flagged. Proposes one focused change, not a list.

→
03 · LAND

Plugin improves itself

The change lands in .claude/rules/, in a plan template, or in a skill's behaviour. Next story benefits — the workflow you ship with tomorrow isn't the one you shipped with today.

Station 08 · Deliver — you

What you get. What stays yours.

By the time /we:story hands off, the PR is in a known-good state. The remaining work — judging it, merging it, closing the ticket — is human work. By design.

What /we:story produces

  • All acceptance criteria implemented and evidenced
  • Tests passing (unit + relevant integration)
  • Code reviewed (code-reviewer agent + CodeRabbit on GitHub)
  • Doc diffs proposed and applied
  • CI green (with CodeRabbit threads resolved)
  • Ticket sitting in „In Review"

What stays yours

  • Reading the PR with your eyes
  • Judging the change — is this still what we wanted?
  • Merging the PR
  • Closing the ticket
  • Communicating the change to stakeholders
  • The decision to ship
Claude never merges PRs or closes tickets. Not a feature flag, not a configuration — a property of the design. Autonomy ends at the gate where consequence begins.
