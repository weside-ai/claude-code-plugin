# /we:docs — Doc Architect

**Role:** Entry point for documentation coherence. Delegates to the
`doc-architect` agent.

**Counterpart:** `/we:arch` (architecture guidance) · `/we:sm` (process).

---

## When to Use

| User intent | Example prompt |
|---|---|
| Find existing docs | `/we:docs Where is memory versioning documented?` |
| Classify new content | `/we:docs I added a dedup pattern for webhooks — where does it go?` |
| Integrate a change | `/we:docs I changed app/db/session.py — what docs need updating?` |
| Audit for drift | `/we:docs Scan architecture/ for drift vs. the code` |
| Regenerate register | `/we:docs Refresh the bypass register` |
| Proactive in pipeline | (invoked automatically by `/we:story` between `simplified` and `pr_created`) |

---

## How It Works

`/we:docs` is **thin by design**. It delegates all the heavy lifting to the
`doc-architect` agent, which:

1. Reads the doc landscape **fresh on every invocation** — rules, indices,
   tree. No cached mental map.
2. Reasons about the user's request in one of 5 modes (question / classify
   / integrate / audit / register). Mode is inferred from the prompt.
3. Produces a concrete proposal (answer with references, classification
   with reasoning, or a diff preview).
4. **Never writes autonomously.** Every file change requires explicit
   approval from the user.

The agent has a hardcoded `writable_paths` allowlist as defense-in-depth:
`docs/architecture/`, `docs/foundations/`, `docs/guides/`, `docs/adr/`,
`docs/vision/`. Writes outside these paths are refused regardless of what
the project's `.doc-architect.yml` says.

## Workflow

### Reactive (user-driven)

1. User runs `/we:docs <question or task>`
1. This skill invokes the `doc-architect` agent via the Agent tool with
   the user's prompt verbatim:

   ```python
   Agent(
       subagent_type="doc-architect",
       description="Doc architect: <short summary>",
       prompt="<user's full prompt verbatim>",
       run_in_background=False,
   )
   ```

1. Agent returns with either (a) an answer, (b) a classification, or
   (c) a proposed diff.
1. If it's a diff: present it to the user, wait for approval, then apply
   the changes via Edit.
1. If the task involved a bypass annotation change: also regenerate
   `docs/architecture/BYPASS-REGISTER.md` via
   `bash scripts/generate-bypass-register.sh --write`.

### Proactive (pipeline-driven, from `/we:story`)

1. `/we:story` calls `/we:docs` between the `simplified` and `pr_created`
   checkpoints with a diff summary
2. Agent returns a list of proposed doc updates (or "nothing needs updating")
3. `/we:story` shows the proposals to the user and waits for approval
4. On approval → apply diffs → write `docs_updated` checkpoint → continue

## Internal Tools (agent can invoke)

In Audit mode, the doc-architect agent **orchestrates `doc-improve`** for the
substantive per-file reviewing. `doc-improve` produces uniform per-file
reports with verdicts, severity-tagged findings, code-grounded evidence,
and (for rules) token budget plus trigger-overlap analysis. The agent
aggregates and prioritises across files; the per-file work is `doc-improve`'s.

- `/we:doc-improve <file>` (or multiple, or glob) → substantive review and
  diff proposals for the named files. Standalone entry point too — when the
  user asks "is this file good?", route to `/we:doc-improve` directly,
  not through this skill.

---

## What This Skill Does NOT Do

- **It does not search the codebase.** The agent reads the landscape fresh
  from rules and the tree, then answers from its mental map. If a concept
  isn't in any doc, it says so explicitly.
- **It does not write autonomously.** Every change is a diff proposal that
  waits for approval. This is a non-negotiable invariant in the agent prompt.
- **It does not replace `/we:arch`.** Architectural guidance (design
  decisions, ADR drafting) lives in `/we:arch`. `/we:docs` handles where
  existing or new content goes in the documentation tree.
- **It does not maintain the bypass register by itself.** The register is
  generated by `scripts/generate-bypass-register.sh`. The agent proposes
  running the generator; the user approves.

---

## Rules

- Always invoke `doc-architect` agent — never inline the logic here
- Never suggest writes outside the agent's writable_paths allowlist
- Always show the agent's diff before applying via Edit
- When called proactively from `/we:story`, keep output concise (just the
  proposal list, no narrative)
- When the agent says "no documentation needed", trust it — do not invent
  work
