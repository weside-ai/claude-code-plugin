# Triage

Move tickets on the project tracker through a small state machine of triage roles — the
systematic intake that turns external signals (bug reports, feature requests, user feedback)
into plan-chain work.

Ticketing runs through the tool-agnostic layer (`references/ticketing.md`) — generic actions,
never tool-specific API calls. When the tracker is GitHub Issues and the maintainer asks for
it, external PRs join the same queue: **a PR is a ticket with attached code** — same roles,
same states, same machine.

Every comment posted to the tracker during triage **must** start with this disclaimer:

```
> *This was generated by AI during triage.*
```

## Roles

Two **category** roles: `bug` (something is broken) and `enhancement` (new feature or
improvement).

Five **state** roles:

- `needs-triage` — maintainer needs to evaluate
- `needs-info` — waiting on reporter for more information
- `ready-for-agent` — fully specified, ready for autonomous pickup (`/we:build`)
- `ready-for-human` — needs human implementation
- `wontfix` — will not be actioned

Every triaged ticket carries exactly one category and one state role. Conflicting state
roles → flag and ask the maintainer before anything else.

These are canonical role names. The actual label strings may differ per repo — read the
optional mapping from `.weside/config.json`:

```json
"triage_labels": { "needs-triage": "bug:triage", "ready-for-agent": "ai-ready" }
```

Absent block or absent key → the label string equals the role name. State transitions: an
unlabeled ticket goes to `needs-triage` first; from there to `needs-info`,
`ready-for-agent`, `ready-for-human`, or `wontfix`; `needs-info` returns to `needs-triage`
when the reporter replies. The maintainer can override any transition — flag unusual ones
and ask.

## Show what needs attention

On "what needs my attention" (or no arguments): query the tracker and present three buckets,
oldest first, with counts and a one-line summary per item — then let the maintainer pick:

1. **Unlabeled** — never triaged.
2. **`needs-triage`** — evaluation in progress.
3. **`needs-info` with reporter activity since the last triage notes** — needs
   re-evaluation.

## Triage a specific ticket

1. **Gather context.** Read the full ticket (body, comments, labels, author, dates; for a
   PR, the diff). Parse prior triage notes so you don't re-ask resolved questions. Explore
   the codebase using the project glossary (`CONTEXT.md`), respecting ADRs in the area. Run
   two checks: (a) **redundancy** — search for an existing implementation of the requested
   behaviour by domain concept, not just the request's wording, and report where you looked;
   found → already-implemented `wontfix` (step 5). (b) **prior rejection** — read
   `docs/plans/out-of-scope/*.md` and surface any file that resembles this request
   ("similar to `out-of-scope/dark-mode.md` — rejected because X. Still feel the same
   way?").
2. **Recommend.** State your category + state recommendation with reasoning and a brief
   codebase summary — including whether it's already implemented. Wait for direction.
3. **Verify the claim.** Before any grilling: for a bug, reproduce it from the reporter's
   steps; for a PR, check out the diff and run the relevant tests. Report confirmed (with
   code path), failed, or insufficient detail (a strong `needs-info` signal). A confirmed
   verification makes a much stronger brief.
4. **Grill (if needed).** If the request needs fleshing out, run `/we:grill` on it — one
   question at a time, sharpening `CONTEXT.md` terms and offering ADRs as decisions land.
5. **Apply the outcome:**
   - `ready-for-agent` — post an agent brief per `references/ticket-briefs.md`.
   - `ready-for-human` — same brief structure, plus why it can't be delegated (judgment
     calls, external access, design decisions, manual testing).
   - `needs-info` — post triage notes (template below).
   - `wontfix` — close, with the comment depending on *why*:
     - **Already implemented** — point to where it lives. Do **not** write to
       `out-of-scope/` — that memory is for *rejected* requests; recording built features
       would poison the dedup check with false rejections.
     - **Rejected (bug)** — polite explanation, close.
     - **Rejected (enhancement)** — write the out-of-scope record (below), link it from the
       closing comment, close.

## The out-of-scope memory — `docs/plans/out-of-scope/`

One file per **concept** (`dark-mode.md`), not per ticket — multiple requests for the same
thing share one file. Readable short-design-doc style: a `# Concept` heading, why it's out
of scope (substantive and durable — project scope, technical constraint, or strategic
choice; never "too busy right now", that's a deferral, not a rejection), and a
**Prior requests** list linking every ticket that asked. On a repeat request: append the
ticket to the existing file's list instead of creating a duplicate. Maintainer changed their
mind → delete the file; old tickets stay closed as historical record. `/we:vision` reads
this directory when sharpening Non-Bets.

## Quick state override

"Move TICKET-42 to ready-for-agent" → trust the maintainer: confirm what you're about to do
(role change, comment, close), then apply directly, skipping the machine. Moving to
`ready-for-agent` without a grilling session → ask whether they want a brief written.

## Needs-info template

```markdown
## Triage Notes

**What we've established so far:**
- point 1

**What we still need from you (@reporter):**
- specific, actionable question 1
```

Capture everything resolved during grilling under "established so far" so the work isn't
lost. Questions must be specific and actionable — "please provide more info" is a no-op.

## Rules

- A resumed ticket starts by reading prior triage notes and checking for reporter answers —
  never re-ask a resolved question.
- Discovery surfaces only *external* PRs (collaborators' in-flight PRs are not triage
  work); an explicitly named PR is always triaged regardless of author.
